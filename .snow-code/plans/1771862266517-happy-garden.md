## Objectives
- Review the existing Planning Poker codebase for bugs, security issues, and performance bottlenecks.
- Identify areas for refactoring and improvement.
- Propose specific fixes and enhancements.

## Exploration Findings
- **Structure**: The project is a ServiceNow application for Planning Poker. It uses Script Includes for backend logic and Fluent (now-sdk) for schema definitions.
- **Key Files**:
    - `src/server/script-includes/*Ajax.js`: Client-callable Script Includes.
    - `src/server/script-includes/PlanningPokerVoteUtils.js`: Utility for vote calculations.
    - `src/server/script-includes/PlanningPokerSecurity.js`: Security logic.
- **Initial Observations**:
    - Code is ES5 compliant.
    - `PlanningPokerVoteUtils.js` has hardcoded T-shirt sizes.
    - Repetitive permission checking logic in Ajax includes.
    - `PlanningPokerSecurity.js` and `PlanningPokerSessionAjax.js` contain potential N+1 query issues and duplicated role logic.
    - **Security Risk**: Potential IDOR in `PlanningPokerStoryAjax.js`.
    - **Security Risk**: Raw error messages exposed to client.

## Specific Issues Identified
1.  **Code Duplication**:
    - `PlanningPokerSessionAjax.js`: `getSession` and `getVotingStatus` duplicate session validation logic.
    - `PlanningPokerSecurity.js` vs `PlanningPokerSessionAjax.js`: Role determination logic is split and duplicated.
2.  **Hardcoded Values**:
    - `PlanningPokerVoteUtils.js`: T-shirt sizes mapped to numbers.
    - `PlanningPokerSecurity.js`: Role names hardcoded.
    - Status values (`'active'`, `'revealed'`) used as string literals throughout.
3.  **Security**:
    - **IDOR Vulnerability**: `PlanningPokerStoryAjax.js` methods (`startVoting`, `setStoryPoints`, etc.) accept `session_id` and `story_id` but fail to verify that the story belongs to the session.
    - **Information Disclosure**: Raw exception objects are returned to the client in error messages.
    - Missing input validation for `story_points` in `setStoryPoints` (allows arbitrary values not in scoring method).
4.  **Performance**:
    - **N+1 Query**: `PlanningPokerSecurity.isUserInVoterGroup` calls `gs.getUser().isMemberOf()` inside a while loop over groups.
    - Missing composite indexes on commonly queried fields (e.g. `session` + `user`).
5.  **Error Handling**:
    - `PlanningPokerVoteUtils.calculateVoteSummary` lacks a try-catch block.
    - Error messages expose internal details.

## Refactoring Plan
1.  **Extract Helper Method**: Create `_validateAndLoadSession` in `PlanningPokerSessionAjax.js` to handle session ID validation and access checks, removing duplication from `getSession` and `getVotingStatus`.
2.  **Optimize Group Check**: Rewrite `PlanningPokerSecurity.isUserInVoterGroup` to fetch all group IDs first, then perform a single `sys_user_grmember` query with `IN` operator to eliminate N+1 queries.
3.  **Centralize Role Logic**: Move the complex role determination logic from `PlanningPokerSessionAjax._determineEffectiveRole` into `PlanningPokerSecurity.getUserRole`.
4.  **Robust Vote Calculation**: Wrap `PlanningPokerVoteUtils.calculateVoteSummary` in a try-catch block and add null checks for input data.
5.  **Fix IDOR Vulnerability**: Update all methods in `PlanningPokerStoryAjax.js` to verify `storyGr.getValue('session') === sessionId`.
6.  **Sanitize Error Messages**: Update catch blocks in all `*Ajax.js` files to log the full error but return a generic user-friendly message to the client.
7.  **Validate Story Points**: Add logic to `setStoryPoints` to verify the value against the allowed `scoring_value` records for the session's method.
8.  **Externalize Constants**: Create `PlanningPokerConstants` script include to house role names, status strings, and T-shirt size mappings.
9.  **Externalize T-Shirt Sizes**: Update `PlanningPokerVoteUtils.js` to use `PlanningPokerConstants.VOTE_VALUES.TSHIRT_MAP`.
